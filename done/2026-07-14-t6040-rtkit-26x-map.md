# T6040 RTKit 26.x firmware/version compatibility map (2026-07-14)

Ticket 028 (offline, P1, cross-cut). Enumerates the RTKit coprocessors on
T6040/J614s, the RTKit version facts shipped in the paired macOS 26.x install,
and the per-driver "add 26.x compat" work — cross-referenced against the Asahi
driver version gates. Static analysis of the paired M4 kernelcache
(`/private/tmp/t6040-ipsw/kernelcache.mac16j.raw`, Darwin 25.5.0
`RELEASE_ARM64_T6041`); no rig, no MMIO. No Apple binary in the repo.

## Headline result

**The RTKit management protocol did not move in 26.x, so no `apple-rtkit` *core*
change is needed.** Compatibility is entirely a per-coprocessor firmware-ABI
question, owned by the (mostly downstream) per-coprocessor drivers. Concretely:

- Asahi `drivers/soc/apple/rtkit.c` gates the management-endpoint **protocol
  version** at `APPLE_RTKIT_MIN_SUPPORTED_VERSION = 11` /
  `APPLE_RTKIT_MAX_SUPPORTED_VERSION = 12`.
- On this 26.x machine, ANS (NVMe) RTKit **already boots** under that unchanged
  core: the storage work reached `RTKit client initialized` and
  `APPLE_ANS_BOOT_STATUS_OK` (`done/2026-07-13-t6040-nvme-map.md`). Since the
  management protocol is uniform across RTKit coprocessors, 26.x reports protocol
  ≤ 12 everywhere. No `MAX_SUPPORTED_VERSION` bump required.
- The RTKit **firmware OS build** shipped in 26.x is `RTKit-1558.40.16.release`
  (string in the kernelcache) — informational; it is not what the core gates on.

So the "26.x compat" work is per-driver firmware-generation/version support, not
an RTKit core patch.

## What actually needs "add 26.x compat" (all [UPSTREAM], drivers downstream)

Ranked by bring-up leverage:

1. **DCP display** (`drm/apple/dcp`) — add the 26.x DCP firmware version to the
   driver's supported set; DCP keys on the exact firmware version, so this is a
   concrete per-version add. → ticket 022. `simpledrm` is the honest interim.
2. **GPU** (`drm/asahi` + Mesa) — G16 (M4) is a new firmware generation; needs
   Mesa `asahi` + kernel `drm/asahi` G16 support (large, upstream long-pole). →
   ticket 039.
3. **SMC** (`mfd/macsmc`, local) — verify macsmc's endpoint/key handling against
   26.x SMC; needed for power button / lid / battery status. → ticket 036.
4. **Audio/SIO** (`apple-admac` + SIO) — SIO firmware protocol version; part of
   the audio path. → ticket 040.
5. ISP / AVD / AVE2 / AOP / PMP / SPU — future; no bring-up dependency.

None of these is an `apple-rtkit` core change; each is a per-driver
firmware-generation/version add in linux-asahi.

## Gaps (what this map cannot pin statically, and why)

- **Exact per-coprocessor firmware build-strings** (e.g. the DCP "Firmware
  Version" that `drm/apple/dcp` keys on). The kernelcache gives generations
  (G16/H16) but not the built firmware version. The staged corpus at
  `/private/tmp/t6040-vendorfw` currently holds **only Broadcom** WiFi/BT (ticket
  014); the Apple RTKit coprocessor firmware needs an `all_firmware.tar.gz`
  extraction — extend ticket 030 to the RTKit coprocessor set.
- **The exact linux-asahi driver version whitelists** (DCP/GPU/ISP supported-fw
  lists). Only the mainline apple drivers are in the local tree
  (`rtkit`/`mailbox`/`sart`/`dart`/`nvme-apple`/`macsmc`/`admac`/`dockchannel`);
  the version-sensitive DCP/GPU/ISP drivers are downstream. Cross-ref the exact
  supported-version arrays against linux-asahi when producing the per-driver
  patches (tickets 022/036/037/039/040).

## Bottom line

The single load-bearing finding is a **negative** one that de-risks the whole
RTKit surface: 26.x keeps the RTKit management protocol ≤ 12, proven live by ANS,
so the shared `apple-rtkit` core needs no work. Every remaining 26.x gap is a
localized, per-driver firmware-generation/version add (DCP version, GPU G16, SMC,
SIO), tracked by tickets 022/036/037/039/040 — not a systemic RTKit blocker.
