# T6040 DockChannel-UART IRQ-816 interrupt-driven path (2026-07-23)

Ticket 062 (offline, P1, console track). Act on the #asahi-dev finding that the
dockchannel-uart AIC input is **816**, not the ADT's 360
(`done/2026-07-21-asahi-dev-irc-review.md`): stand up an interrupt-driven
`ttydc0` and retest whether RX actually fires — the open question left by the
rxirq-txpoll diagnostic (`total=0`, NEXT_STEPS #0), which listened on input 360.

## DT state — most of the ticket was already in place

Reviewing `dts/t6040-j614s-dcuart.dts` (the base, poll-mode console):

- **IRQ 816 already set** (`interrupts = <AIC_IRQ 816 IRQ_TYPE_LEVEL_HIGH>`), so
  the ADT-360 error is already corrected in our DT.
- **RX/TX bits are already DT properties**: `apple,irq-rx-mask = <0x2>` (BIT(1),
  the correct UART RX bit) and `apple,irq-tx-mask = <0x4>` (BIT(2)). This is the
  "make the bits a DT property" shape the upstream discussion (yuka/integralpilot)
  converged on — we already have it. MTP uses BIT(3) (`0x8`), set separately on
  `&mtp_dockchannel`, so the two instances are correctly distinguished.
- **reg-names** already expose `config @0x508828000` and `data @0x50882c000`
  (node unit-addr `0x50880c000` is the irq block) — matching the IRC-reported
  addresses. Our stack is mailbox (`apple,t8122-dockchannel`) → serial child
  (`apple,dockchannel-serial`), which consumes data via the mailbox rather than
  an OF earlycon on reg[0]; so the "data reg must be reg[0] for earlycon" point
  applies to the *direct* `apple,dockchannel-uart` driver, not our split stack.
- The only thing forcing the polled path is **`apple,poll-mode;`**.

## The variant this ticket adds

`dts/t6040-j614s-dcuart-irq816.dts` = the base dcuart DT **minus `apple,poll-mode`**
— interrupt-driven RX/TX on input 816, everything else identical. This is the
minimal change that answers "does IRQ-driven RX fire on the correct line?".

Built + verified (kbuild container, kernel dtc): the `mailbox@50880c000` node
decompiles to `interrupts = <0x0 0x330 0x4>` (0x330 = 816, LEVEL_HIGH),
`apple,irq-rx-mask = 0x2` (BIT1), `apple,irq-tx-mask = 0x4` (BIT2), reg-names
irq/config/data, and **no `apple,poll-mode`**. DTB SHA-256
`21a3446830a50a0d739a6e55e2ef9f9ee3986bcc677d12e5caceced708cf22a2`. dtc-clean.

## Driver audit — the interrupt path is already supported by the build stack

The committed in-tree mailbox driver (`drivers/mailbox/apple-dockchannel.c`)
hardcodes `IRQ_RX = BIT(3)` and has **no** poll-mode or configurable masks — so
on its own it would drive the UART RX on the wrong bit. The bring-up behavior
comes from patches the standard `DOCKCHANNEL=1` build applies, in order
(`scripts/t6040-kbuild.sh` ~L618–650):

1. DockChannel serial TTY (`apple,dockchannel-serial` → `/dev/ttydcN`), applied
   from a pinned git blob `b8dcbdcb…` (this is why it isn't a `.patch` file).
2. **`t6040-dockchannel-poll.patch`** — adds `apple,poll-mode` and makes the RX/TX
   IRQ bits DT-configurable via `apple,irq-rx-mask`/`apple,irq-tx-mask`
   (so our `0x2`=BIT(1) UART RX is honored); when `apple,poll-mode` is **absent**
   it takes the interrupt path (`platform_get_irq` + threaded IRQ, `enable_irq`
   in `.startup`).
3. **`t6040-dockchannel-rx-rearm.patch`** (2026-07-23) — masks RX locally before
   the write-1-to-clear ack, then re-enables and re-drains in the threaded
   handler (`drain:` loop). This closes exactly the ack-vs-drain race flagged in
   NEXT_STEPS #0 (driver W1C-acking RX while it stays unmasked → storm), so the
   interrupt path is safe to exercise.

**Conclusion: no new kernel patch is needed.** The standard dcuart kernel built
with `DOCKCHANNEL=1` already contains the mask-configurable IRQ path *and* the
rx-rearm fix. The IRQ-816 variant DTB (no `apple,poll-mode`) selects the
interrupt path; together they are the test. The one requirement is that the
kernel `Image` used for the run be built **after** the rx-rearm patch existed
(2026-07-23) so it includes the ack-race fix.

## Rig test (the go/no-go)

Boot the standard dcuart kernel (`DOCKCHANNEL=1`, incl. rx-rearm) + the IRQ-816
no-poll-mode DTB + the dcuart initramfs. The poll-mode base DT already gives an
interactive `ttydc0` shell that responds to typed input **via polling**; this
variant drives RX via **AIC input 816 interrupts** instead. Pass/fail:

- **Pass:** the `ttydc0` shell still echoes/responds to typed host input (RX
  delivered through the IRQ), and no IRQ storm (watchdog/heat) occurs — i.e. the
  rxirq-txpoll `total=0` result is resolved by the corrected input.
- **Fail (no response):** interrupt-driven RX still not delivered on 816 → escalate
  (compare the exact AIC mask-write timing against yuka's working M4 Pro build).

Optionally add the telemetry patch (`apple,irq-telemetry`) to read handler-entry
and virq→hwirq counts, but the interactive echo is a sufficient first go/no-go.

## Status of the console today (context)

The poll-mode base DT already gives a working two-way `ttydc0` shell over
DebugUSB (DEVLOG), so this is **not** a fix for a broken console — it closes the
interrupt-delivery question and is the prerequisite for a non-busy-polling /
printk-capable console (the parked console-safety item). Poll-mode stays the
default and fallback.
