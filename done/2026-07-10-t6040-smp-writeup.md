# m1n1 SMP/WFI report for T6040 (M4 Pro)

**Ready for CJ to post to `#asahi-dev` (finalized 2026-07-23).** The text in
the next section is the posting draft. Nothing has been posted externally.

## Posting draft

I have m1n1 SMP execute-and-return working reliably on a tethered T6040
(Mac16,8/J614s, M4 Pro). The boot CPU is a P-core (`smp_id 4`); all 14 active
cores start (4E + 5P + 5P). `CPU_START_OFF_T6031` (`0x88000`) is correct for
Brava. The J614s ADT has no cpu-id 9, which accounts for the gap between the two
P clusters.

The measured MPIDRs are:

```text
E cluster:  smp_id 0-3    MPIDR 0x80000000-0x80000003
P cluster0: smp_id 4-8    MPIDR 0x80010100-0x80010104
P cluster1: smp_id 10-14  MPIDR 0x80010200-0x80010204
```

For an execute-and-return check I uploaded
`add x0, x0, x1; ret`, performed D/I cache maintenance, and invoked it with
`smp_call_sync` on an E and a P secondary. It returned `deadbeef` and
`11112222` for the two test vectors. All 13 secondaries report alive and every
active core returned the expected per-core heartbeat.

One wrinkle is WFI retention on raw-boot T6040. Sven's `c22ca847`
(`chickens: clear CYC_OVRD_DISABLE_WFI_RET`) identifies why a secondary can lose
most of its register file and re-enter at RVBAR after a stop/start cycle: the
disable-retention bit can remain set. That fix clears the bit through
`SYS_IMP_APL_CYC_OVRD`, but m1n1 deliberately performs that register access only
when `apple_sysregs_unlocked` is true. T6040/T6041 remain locked on current
firmware, so the fix is a no-op on this raw-boot machine.

I therefore have a small fallback patch which marks the M4 feature set
`broken_wfi`, parks secondaries in WFE from first start, and refuses a proxy
request that would switch them back to `deep_wfi()`. This has remained stable
across repeated proxy sessions and kernel boots. Yuka's `2e6ac0b0` WFI/WFIT
experiment is useful coverage for the EL1/EL2 register-retention behavior.
Sven's separate `954f80c6` secondary-MMU barrier/stack-cache fix is also in my
test tree; it fixes a distinct stale-stack problem and does not make the locked
`CYC_OVRD` path available.

Linux still boots here with `idle=nop`, and the locked-AIC sysreg avoidance
remains required. EFI-PSCI CPU power-down looks like the eventual standard
path, but it does not replace this raw-kboot/WFE fallback today.

I can send the `broken_wfi` patch and the complete J614s MPIDR/session log if
that shape is useful, or retest a narrower workaround suggested by maintainers.
